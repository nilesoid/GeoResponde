# Scientific Intelligence Layers: Source Inventory

This document catalogs the "Scientific" category layers exposed by GeoResponde Venezuela and the real data sources behind them. It was generated by reading `public/catalog/layers.json`, `public/catalog/datasets.json`, and `public/catalog/sources.json` directly, and then cross-checking every endpoint against the backend adapter code in `backend/src/adapters/` (specifically the `usgs`, `funvisis`, `eonet`, and `damage` adapters, plus `packages/shared/src/eventRegistry.ts` where the actual NASA FeatureServer URL lives). Every value below reflects what is actually written in code today, not what the catalog metadata implies should be there. Where the catalog and the adapter disagree, both are noted.

Nine layers in `layers.json` carry `category: "Scientific"`: recent earthquakes (USGS), FUNVISIS local earthquakes, tectonic faults (GEM), USGS geologic units, the Esri "before" satellite basemap, two Copernicus EMS damage layers, and two NASA damage/interferogram layers. Walking the graph from each of those layers through their `datasetIds` and each dataset's `sourceId` resolves to seven distinct, non-placeholder sources: USGS (earthquakes and geology), FUNVISIS, GEM, Copernicus EMS, NASA, and Esri (World Imagery). One more source, Vantor Open Data, exists in `sources.json` but is not currently wired to any dataset, which is called out below.

## Notes on individual sources

**USGS earthquakes** is the most solid entry in the whole inventory: a public-domain, globally scoped federal feed with no authentication and an hourly refresh promise in the catalog. It is worth knowing the catalog's `sources.json` entry points at the static `2.5_day.geojson` summary feed, while the actual backend service builds its own query against the fdsnws `/query` endpoint with bounding-box and start-date parameters. Both URLs are real USGS endpoints; the code just uses the more flexible one.

**FUNVISIS** shows the same kind of split. The catalog lists FUNVISIS's own `maravilla.json` feed as the source, but the adapter that ships in `backend/src/adapters/funvisis/service.ts` never touches that URL. It fetches from SismosVE, a third-party mirror of FUNVISIS data, and applies its own date filtering client-side since that mirror has no server-side date parameter. This layer is also disabled by default in the catalog (`enabled: false`) and marked with a `Static` refresh frequency rather than a real polling interval.

**GEM Global Active Faults** is a static academic dataset distributed as a GitHub repository rather than a live API. There is no endpoint to poll; consuming it means pulling a snapshot of the shapefile/GeoJSON data GEM publishes and serving it locally. Its CC-BY-SA-4.0 license is one of the only explicitly stated licenses anywhere in the catalog.

**USGS Data Series 199** is a published geologic map product (a PDF/report with accompanying geologic polygon data), not a queryable API. It now backs only the geologic-units layer.

**Esri World Imagery** is the "before" satellite basemap. It is served as raster tiles from the Esri ArcGIS Online World Imagery service (`https://server.arcgisonline.com/ArcGIS/rest/services/World_Imagery/MapServer`). This used to be a data inconsistency: `ds-satellite-before` referenced `src-usgs-ds-199` (the USGS geologic map) even though its title and description describe Esri imagery. That was resolved under issue #46 by adding a dedicated `src-esri-world-imagery` source (under a new `org-esri` organization) and repointing the dataset to it. The Vantor Open Data Program source (`src-vantor`, https://vantor-opendata.s3.amazonaws.com) still sits in `sources.json` without any dataset wired to it; it is a separate provider (unrelated to Esri) kept for potential future imagery use, and this change deliberately does not touch it.

**Planet Labs PBC via Source Cooperative** provides post-event satellite imagery (`ds-satellite-after`). GeoResponde consumes the imagery through Source Cooperative in accordance with the project's federated architecture and does not maintain local copies unless technically required.
**Copernicus EMS** damage and ground-movement layers are per-event products tied to a specific EMSR activation code (EMSR884 in the seeded event data), not a continuously polled feed. The backend hits the public Copernicus Rapid Mapping dashboard API for that one activation and merges whichever damage-grading and ground-movement layers it returns. Everything here is disabled by default and labeled `Static`, which matches how EMS actually works: one map product per disaster, not a rolling update.

**NASA Sentinel damage proxy and interferogram** layers carry an explicit disclaimer in the catalog ("Experimental, not validated for operational use"), and the backend code treats the underlying ArcGIS FeatureServer as a huge (2.7 million polygon) anonymous public layer that needs careful server-side filtering and pagination to stay usable. Like Copernicus, this is a per-event product scoped to one Sentinel-1 pass over one disaster, not a live global feed.

**OpenStreetMap Overpass** (`src-osm-overpass`, https://overpass-api.de) does appear in `sources.json`, but it only feeds `dataset-hospitals`, which is tagged `Humanitarian`, and no layer in `layers.json` currently references that dataset at all. It is not part of the Scientific-category graph and is mentioned here only for completeness.

A number of other entries in `sources.json` (shelters, road closures, damage assessment, satellite imagery, landslides, power outages, airports, ports, volunteer organizations, donation campaigns, aftershocks, and the Chile collection centers) point at `https://example.com/placeholder` and are unfilled stubs, not real documented data sources. They were excluded from the table above entirely.

## What is genuinely unknown

License is unknown for FUNVISIS, USGS Data Series 199, Copernicus EMS, and NASA EarthData because `sources.json` does not carry a `license` field for those entries and no license statement was found in the adapter code or comments. No numeric refresh cadence (in minutes or hours) was found for anything beyond the "Hourly" value on the USGS layer; every other Scientific layer is marked `Static` in the catalog itself, which is reported as-is rather than guessed at.

## Contributor research template

If you are helping research a new source and do not want to touch GitHub directly, fill one block per source with exactly these fields and hand it back. That maps one to one onto the table above, so it drops straight in.

```
Source name:
Category:            (Earthquakes / Fire / Flood / Damage / Satellite imagery / Humanitarian / other)
API:                 (the endpoint URL or the access method, e.g. "FDSN event web service at <url>", "STAC over S3", "cdsapi client")
License:             (e.g. Public Domain, CC-BY-4.0, Copernicus, "unknown")
Update frequency:    (e.g. real-time, hourly, daily, per event, static)
Geometry:            (points / lines / polygons / raster)
Coverage:            (global / Venezuela / per-event / a named region)
Authentication:      (none / free API key / account required / "unknown")
Notes:               (anything worth knowing: rate limits, per-event scoping, experimental status)
```

Field notes so everyone fills them the same way:

- **API**: prefer the real machine-readable endpoint (a web service or data file), not just the human website. If the only access is a web page, say so.
- **Authentication**: "none" means you can call it with no key. If a free key or account is needed, write that plus where to get it.
- **Geometry**: what shape the data is. Point feeds (earthquakes, fires), vector polygons (damage grading), or raster tiles (satellite imagery, flood grids).
- **Coverage**: global, a single country, or scoped to one disaster activation at a time.
- If you are unsure of a field, write "unknown" rather than guessing. A verified "unknown" is more useful than a wrong value.
