# Humanitarian Provider Catalog

This is an inventory of the humanitarian provider platforms that GeoResponde's backend adapters federate. It was generated by reading the federation adapter code under `backend/src/adapters/` and the catalog data in `public/catalog/`, current as of 2026-07-02. Most of these platforms sprang up in response to the 2026 Venezuela earthquake, so the default geographic scope is Venezuela. Two of them (Humanitarian Data Exchange and Ushahidi) are general humanitarian platforms with global scope, and they are flagged as such below.

Each row lists what the adapter code actually does: the endpoint it talks to, whether it reads JSON, scrapes HTML, or pulls a framework payload, and the authentication it sends. Where a field cannot be determined from the code or catalog, it says "unknown" rather than a guess. The Organization column comes from `public/catalog/organizations.json`; only HDX has an explicit organization mapping there, so the other providers are marked "unknown".

## Notes by provider

### Refugios Venezuela
This entry is a test placeholder, not a working integration. Its adapter is `MockHumanitarianAdapter`, which lives in `backend/src/testing/MockHumanitarianAdapter.ts` and returns hardcoded data without any network call. It is not registered in `backend/src/adapters/registry.ts`, so in production `createAdapter()` would return nothing for `prov-refugiosvenezuela`. It appears in `public/catalog/providers.json` but is effectively inert outside tests.

### Úbícame
The dataset is partitioned into 26 static JSON shards named A through Z by the first letter of the person's name. The adapter fetches the relevant shard directly. There is no per-person detail endpoint.

### Venezuela Reporta
Reads are unauthenticated. A live POST only fires when three conditions all hold: `opts.dryRun === false`, the environment flag `GEORESPONDE_SUBMIT_LIVE === '1'`, and an API key is present in `VENEZUELAREPORTA_API_KEY` (sent as the `x-api-key` header). Otherwise the submission stays a dry run. The provider documents a 120 requests/minute rate limit and requires attribution. Cédula is sent as part of a consented report but is never logged or previewed (it is stripped by `redactSubmissionBody`).

### Desaparecidos Terremoto VE
Only the directory of aid platforms is federated. A code comment states the people/missing-persons endpoint is protected by reCAPTCHA v3 and cannot be federated server to server. The submission path is an ethical deep link to the provider's own form (no network call, no scraping), with a placeholder mailto contact.

### Hazlo Hoy Terremoto
The site has no open JSON API. The adapter requests the homepage with a `RSC: 1` header, receives a Next.js React Server Component payload (`text/x-component`), and extracts the embedded `markers` array. Parsing uses manual bracket balancing and escape normalization because the array is embedded inside the streamed payload rather than served as clean JSON.

### Reencuentra VE
There is no public JSON API (the site's Supabase backend is closed), so the adapter scrapes the search results HTML with cheerio. It only returns the first page, up to 20 cards, because real pagination is driven by a Next.js Server Action that is not present in the static HTML.

### Busca NexoSignal
The Supabase anon key is hardcoded in the adapter source. A code comment explains this is the public, embeddable anon-style key that the site itself ships to the browser, so it is considered safe to keep in source. A separate comment warns that the `cedula_reporta` field is the ID of the person filing the report, not the found child, and must never be mapped to `person.cedula`.

### Terremoto Venezuela
This source also stores names of trapped or missing people. A code comment states those are never requested or surfaced; only a `has_missing_persons` flag is exposed. Submission of building-damage reports is dry-run by default and only goes live when `GEORESPONDE_SUBMIT_LIVE === '1'`. All mapping lives inline in the adapter (there is no separate parser file).

### Sismo Venezuela
A code comment describes this as a community platform that aggregates crowdsourced earthquake-damage reports scraped from social media (YouTube, X, Instagram). The original source does the scraping; GeoResponde's adapter only consumes the processed JSON feed. There is no server-side search parameter, so filtering happens client side, and results carry a verified or unverified marker.

### Ayuda Venezuela
The Supabase anon key is hardcoded, and a code comment warns it is extracted from the public site bundle and rotates between deploys of ayudavenezuela.app. If the adapter starts returning 401 or empty results, the fix is to re-extract the current publishable key from the site's JavaScript bundle and update the constant.

### Venezuela Reporta (Ingresos)
This reads a hospital-intake roster of roughly 20,000 admitted persons. A code comment stresses that no status is emitted, because appearing in an intake roster does not confirm a person is safe. Attribution to Venezuela Reporta is required.

### Ushahidi
Submission-only: `search()` always returns an empty array. A live POST requires `GEORESPONDE_SUBMIT_LIVE === '1'` plus three environment variables: `USHAHIDI_TOKEN` (OAuth2 Bearer token), `USHAHIDI_DEPLOYMENT_URL` (the target deployment), and `USHAHIDI_FORM_ID`. The README notes the v5 API contract was verified against the `ushahidi/platform` source code rather than the docs SPA. Fields marked sensitive (such as cédula) are never sent to a third-party public intake.

## Scope note on disaster and risk data services

The `backend/src/adapters/` directory also contains `funvisis`, `usgs`, `eonet`, `damage`, and `sitios`. These are not part of the person-search provider federation and are not listed in `public/catalog/providers.json` or `registry.ts`. They feed geospatial risk and disaster layers for the situation map (seismic feeds, USGS and NASA EONET event data, Copernicus EMS and NASA ARIA damage mapping, and a sites layer). They use a different architecture (`service.ts` with TTL cache and safe degradation) instead of the `BaseAdapter` federation pattern, so they are out of scope for this provider inventory.
